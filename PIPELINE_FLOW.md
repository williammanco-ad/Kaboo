## ğŸ¯ Key Decision Points

1. **When to commit**: Anytime during development
2. **When to push**: When feature is complete
3. **When to tag**: When ready for release
4. **When to deploy to TestFlight**: For beta testing
5. **When to deploy to App Store**: For production release
6. **When to submit for review**: After metadata is complete

## ğŸ“ Command Reference

| Action | Command | Trigger |
|--------|---------|---------|
| Development | `git push origin main` | CI workflow |
| Create Release | `git tag v1.0.0 && git push origin v1.0.0` | Release workflow |
| TestFlight | GitHub Actions UI or `fastlane beta` | App Store workflow |
| Production | GitHub Actions UI or `fastlane release` | App Store workflow |
| Version Bump | `./scripts/version.sh patch` | Creates tag locally |

## ğŸ”„ Typical Release Cycle

1. **Week 1-2**: Development
   - Push code regularly
   - CI runs on each push
   - Tests run automatically

2. **Week 3**: Beta Testing
   - Tag version: `v1.0.0-beta.1`
   - Deploy to TestFlight
   - Collect feedback from testers

3. **Week 4**: Production Prep
   - Fix beta feedback issues
   - Tag production version: `v1.0.0`
   - Deploy to App Store
   - Submit for review

4. **Week 5**: Release
   - App approved by Apple
   - Available on App Store
   - Monitor crash reports

5. **Ongoing**: Maintenance
   - Patch releases: `v1.0.1`, `v1.0.2`
   - Minor updates: `v1.1.0`
   - Major updates: `v2.0.0`
